## 1. 复审结论

本轮整改包含 8 个提交，涉及约 50 个文件，已经从修复少量 P0 缺陷扩展为月度计划、长期项目、公司规则、AI 世界模型、证据元数据、价值目标、危机章节和批量模拟等系统性建设。

整改不是表面换文案。以下问题已经得到实质修复：

- 金额字段不再统一限制到 `0..100`；
- 投资收益不再同时增加现金和投资组合；
- 求职待处理次数与终身累计次数已经分离；
- 无伴侣和无子女状态不会再直接生成相应支持与育儿压力；
- 依赖版本已经锁定；
- 年龄摩擦开始被表达为外部市场和组织偏见；
- 心理健康和健康危机不再第一次触底就立即结束游戏；
- 测试与批量模拟框架明显扩展。

但当前版本仍然存在明显的“系统骨架已创建，玩家玩法未接通”问题：

1. 月度计划引擎支持多行动，但界面仍然每月只执行一个行动；
2. 危机能够进入，却不能真正恢复、关闭或重新开启；
3. 新职业路线、公司薪资和晋升参数大量停留在配置层；
4. 项目进度隐藏且只能完成一次；
5. 证据元数据存在错误分类，仍通过正则猜测自由文本条件；
6. 固定价值权重没有玩家选择过程，却被界面标记为“价值匹配”；
7. 旧的单调 AI 替代指标仍继续影响工资与行动收益；
8. 发布门槛过松，所谓“结局可达性”并没有验证真实游戏路径。

## 2. 复审判定标准

本报告不以新增文件或新增类型作为“完成”依据。每项整改必须同时满足：

1. **玩家可达**：玩家能从 UI 实际使用该功能；
2. **规则接入**：配置真正参与工资、事件、职业、结局等核心公式；
3. **状态闭环**：系统具有进入、运行、退出、失败与存档恢复路径；
4. **反馈一致**：UI 文案、可见数值和实际效果一致；
5. **测试有效**：测试从合法初始状态验证真实转移，而不是直接伪造终态；
6. **数据可追溯**：外部依据包含类型、链接、日期、范围与参数映射理由。

状态定义：

- **已修复**：核心行为闭环并有有效测试；
- **部分修复**：底层或部分场景完成，但玩家不可达、存在旁路或测试不足；
- **未修复**：核心问题仍然存在；
- **新问题**：整改过程中新增的行为错误或架构矛盾。

## 4. 高优先级问题

### FOLLOWUP-001：月度多行动系统没有交付到 UI

**级别：P0**

**证据**

- `src/core/gameEngine.ts` 新增 `planMonth(state, actionIds)`；
- `src/App.tsx:307` 通过 `calculateMonthlyPlanBudget(state)` 计算空计划预算；
- `src/App.tsx:404` 点击行动时仍执行 `planMonth(state, [action.id])`。

**实际行为**

界面显示“月度计划”“时间预算”和“精力预算”，但没有：

- 已选行动列表；
- 添加与取消操作；
- 本月计划确认按钮；
- 单项时间和精力成本；
- 超载前的实时反馈。

每次点击仍然立即执行一个行动并推进月份。预算面板中的 `used` 来自新建空计划，因此始终为 0。

**影响**

这不是轻微 UI 缺失，而是核心玩法仍停留在旧循环。引擎测试证明“API 可以多行动”，不能证明玩家已经获得多行动玩法。

**修复方案**

在 `GameScreen` 增加本地计划状态：

```ts
const [plannedActionIds, setPlannedActionIds] = useState<string[]>([]);
```

行动按钮改为选择/取消，不立即推进月份；预算预览使用 `buildMonthlyPlan(state, selectedActions)`；新增“执行本月计划”命令。

同时必须处理：

- 禁止重复行动 ID，除非行动明确允许多次；
- 冷却状态在选择和执行时二次验证；
- 存在 `pendingEventChoice` 时禁止提交新计划；
- 多月行动必须锁定后续时间，或明确一次性跳过多个月；
- 计划执行后清空选择；
- 移动端固定高度区域不能遮挡提交按钮。

**验收标准**

- 玩家可以选择至少两个行动并一次结算；
- UI 实时显示已使用预算；
- 超载计划无法提交；
- 单行动兼容接口不再作为主要 UI 路径；
- E2E 测试从点击两个行动到月份推进完整通过。

### FOLLOWUP-002：危机章节没有恢复、关闭和复发机制

**级别：P0**

**证据**

- `CrisisChapter` 包含 `active`、`startedMonth`、`recoveryProgress`；
- `endingSystem.activateCrisis()` 只负责写入 `{ active: true, recoveryProgress: 0 }`；
- 没有月度系统增加 `recoveryProgress`；
- 没有任何逻辑将危机恢复为 `active: false`；
- 硬失败只检查 `active` 和开始时间是否超过六个月。

**影响**

第一次燃尽后即使精神完全恢复，危机仍永久激活。数年后再次跌到阈值时，旧的 `startedMonth` 已经超过六个月，玩家可能立即失败，得不到新的恢复窗口。

`recoveryProgress` 当前只是未使用字段。文案表达“可以恢复”，状态机并没有实现这一承诺。

**修复方案**

新增 `crisisSystem.ts`，每月在结局检查前结算：

```text
inactive -> active -> recovering -> recovered
                         -> failed
```

恢复进度由以下因素决定：

- 睡眠和休息；
- 心理咨询或医疗支持；
- 工作负荷下降；
- 现金缓冲；
- 家庭与朋友支持；
- 边界和健康状态。

恢复后保存 episode 历史并关闭 `active`。复发必须创建新的 `startedMonth`，同时可以保留复发敏感性，而不能沿用第一次倒计时。

**验收标准**

- 危机经过有效恢复后会关闭；
- 恢复后不会因为旧时间戳立即失败；
- 第二次危机拥有独立恢复窗口；
- 存档恢复不会丢失危机阶段；
- 测试覆盖恢复、失败、复发和跨版本存档。

### FOLLOWUP-003：职业路线和公司配置没有进入核心玩法

**级别：P0**

**证据**

- `src/config/realworldCareer.ts` 定义九条 `CAREER_ROUTES`；
- `CareerState.track` 仍只允许前端、后端、全栈、AI应用四类；
- 没有职业转型行动、路线状态和 UI；
- `CompanyProfile.salaryCoef` 和 `promotionSpeed` 没有被工资与晋升公式使用；
- `getGrossSalary()` 仍使用固定 `JOB_LEVELS`、城市和经济周期；
- `CAREER_ROLES`、真实城市数据与真实公司数据主要由加载测试使用。

**影响**

玩家无法体验测试、数据工程、安全、嵌入式、SRE、管理或行业专家路线。公司配置虽然丰富，但工资和晋升仍没有形成公司差异。

**修复方案**

明确选择一种方案：

1. 将扩展路线正式接入 `CareerState` 和转型系统；
2. 或暂时移除不可玩配置，避免制造已完成功能的错觉。

推荐增加：

- `currentRoleId`；
- `roleHistory`；
- `transitionProgress`；
- `domainExperience`；
- `transferableSkills`。

工资应从真实角色、城市、公司类型和级别匹配中生成；晋升速度使用公司 `promotionSpeed`，学习速度使用 `learningTime`，薪资使用 `salaryCoef`。

**验收标准**

- 至少三条新增路线可以从 UI 完成转型；
- 公司薪资和晋升参数产生可测差异；
- 未接入的数据不得在版本说明中称为已支持功能；
- 模拟策略可以主动选择职业转型。

### FOLLOWUP-004：项目系统隐藏、单实例且结果说明错误

**级别：P0**

**证据**

- `ProjectPortfolioState` 对每类活动只保存一个 `ProjectState`；
- `projectSystem.ts:33` 在项目完成后直接返回，不能创建下一项目；
- UI 没有读取 `state.projects`；
- 行动摘要仍显示“作品 +1”“被动收入”，但实际已被延迟到完成；
- 即时 XP、精神和健康变化仍然保留，只有部分最终收益被延迟。

**影响**

玩家不知道项目为何没有立即获得作品，也不知道还差多少完成。完成第一个项目后，项目实战、写作和开源的长期项目系统永久失效。

**修复方案**

将单例状态改为项目实例：

```ts
interface ProjectInstance {
  id: string;
  kind: ProjectKind;
  status: 'planned' | 'active' | 'released' | 'abandoned';
  progress: number;
  quality: number;
  audienceFit: number;
  startedMonth: number;
  completedMonth?: number;
}
```

项目完成后进入历史记录，玩家可以创建新项目。UI 展示进行中项目、完成度、质量、预计收益和是否发布。

**验收标准**

- 玩家能完成两个连续的项目实战；
- 项目进度在 UI 可见；
- 行动摘要与实际即时/延迟效果一致；
- 作品质量影响声望、求职和副业结果；
- 项目支持暂停和放弃，而不是只有完成。

### FOLLOWUP-005：证据治理存在错误分类和自由文本猜测

**级别：P0**

**证据**

- 所有现实行动与事件几乎统一标记为 `industry_report`；
- `EvidenceMetadata` 只保存来源等级、置信度和可选来源名称；
- `source_url`、日期、样本、适用人群和参数理由没有进入元数据；
- `requirementsFor()` 使用正则匹配中文自由文本；
- 部分解析成功不代表全部条件都被识别；
- 测试只检查元数据枚举合法，不验证来源是否真实或分类正确。

**影响**

社区帖子、媒体文章和研究报告可能被同样标记为行业报告。只要填写一个允许的枚举，测试就会通过。这会把“存在来源字段”误当成“证据可信”。

**修复方案**

扩展元数据：

```ts
interface EvidenceMetadata {
  sourceType: 'peer_reviewed' | 'official_statistics' | 'industry_survey' | 'media' | 'community_story' | 'synthetic';
  title: string;
  url?: string;
  publicationDate?: string;
  population?: string;
  sampleSize?: number;
  applicableScope: string[];
  confidence: 'high' | 'medium' | 'low' | 'narrative_only';
  parameterRationale?: string;
  verifiedAt?: string;
}
```

行动条件直接在源 JSON 中保存结构化 `requirements`。自由文本仅用于展示，不参与规则解析。未迁移条件必须在构建阶段报错。

**验收标准**

- V2EX 等社区内容标记为 `community_story`；
- 社区故事默认不能直接校准核心概率；
- 每个用于数值校准的来源有 URL 和参数理由；
- 不再通过正则从中文描述生成规则；
- 构建时验证所有条件和证据元数据。

## 5. 中优先级问题

### FOLLOWUP-006：事件链次数与冷却时间共用 `eventMemory`

**级别：P1**

普通事件触发时先对 `eventMemory[event.chain]` 加一，随后又将 `eventMemory[event.cooldownKey]` 写成当前月份。默认冷却键可能就是 `event.chain`，导致计数被时间戳覆盖。

建议拆分：

- `eventChainProgress`；
- `eventLastTriggeredMonth`；
- `eventChoiceMemory`；
- `eventFlags`。

互斥标签也不应在候选池生成阶段优先保留最高强度事件。当前每月只抽一个普通事件，过早删除同标签候选会降低事件多样性。

### FOLLOWUP-007：价值目标不是玩家价值

**级别：P1**

`DEFAULT_PLAYER_VALUES` 给所有玩家相同权重，界面没有选择或形成过程。`deriveLifeSatisfaction()` 又没有使用这些权重，却在 UI 中显示为“价值匹配”。

建议：

- 开局只选择 2～3 个初始倾向，而不是填写八个滑块；
- 重大选择逐步改变价值权重；
- 结局展示“玩家选择形成的价值”与结果匹配，而不是默认模板；
- `deriveLifeSatisfaction()` 与 `calculateValueFit()` 合并为一个权威实现；
- 压力变化必须使用同一指标计算 delta。

当前 `PressureSummary` 的“生活满意”值使用 `100 - lifeSatisfaction`，但 delta 仍绑定旧 `marketPressure` 键，变化提示没有相同语义。

### FOLLOWUP-008：旧 AI 替代模型仍在影响核心结果

**级别：P1**

新模型增加了模型能力、采用率、组织准备度、信任和岗位自动化率，但旧 `aiReplacement` 仍每月单调增长，并继续降低工资和成长行动收益。

建议：

- 将 `aiReplacement` 降级为兼容字段或彻底删除；
- 工资、求职、裁员、行动收益统一使用 `deriveRoleAiPressure()`；
- `taskAutomationByRole` 应随模型能力、企业采用和监管变化，而不是永久固定；
- 将 AI 能力拆成使用、验证、集成、问题定义与治理能力。

### FOLLOWUP-009：家庭责任和年龄成本仍然自动增长

**级别：P1**

无伴侣、无子女的直接矛盾已经修复，但：

- `familyResponsibility` 仍然随年龄为所有玩家增长；
- `getMonthlyCost()` 在 32 岁后每年自动增加 70 元；
- 默认 `hasParents=true`，玩家无法配置家庭背景；
- `relationship_bankrupt` 结局没有检查玩家是否存在伴侣或家庭承诺。

年龄可以改变外部环境和健康恢复，但不能自动等于家庭责任和生活成本。应由住房、赡养、子女、医疗和玩家选择形成具体支出。

### FOLLOWUP-010：商店固定支出没有按组成项结算

**级别：P1**

购买商业医疗险会增加 `monthlyFixedCost` 80 元，但 `settleFixedFinance()` 使用：

```ts
Math.max(getMonthlyCost(state), state.finance.monthlyFixedCost)
```

在城市生活成本高于 `monthlyFixedCost` 时，新增的保险费用会被吞掉。

应将固定成本拆成基础生活、租金、订阅、保险、债务、家庭支出和交通，然后求和。禁止用 `max()` 合并不同费用来源。

### FOLLOWUP-011：多行动 API 存在重复执行与冷却旁路

**级别：P1**

`planMonth()` 在执行前统一检查原始状态，没有验证行动 ID 唯一，也没有在每个行动执行前重新检查冷却和条件。

调用方可以提交重复行动；第一个行动写入冷却后，第二个相同行动仍可能执行。现实数据行动虽然设置 `cooldownMonths`，其 `require` 主要只检查结构化需求，并不统一检查冷却。

建议建立统一 `canExecuteAction(state, action, currentPlan)`，UI、模拟器和引擎全部复用。

### FOLLOWUP-012：派生指标仍然存在双重真相

**级别：P1**

系统同时保存：

- `stats.burnout` 和派生燃尽风险；
- `careerProfile.employability` 和再次派生的可雇佣性；
- `healthProfile.healthDebt` 和再次派生的健康债；
- `stats.relation`、关系债、孤独与安全网。

新增派生层是正确方向，但应明确哪些是源状态，哪些是计算结果。派生指标不应继续作为可直接加减的持久状态。

### FOLLOWUP-013：成就和结局条件与文案不一致

**级别：P1**

“职业转型窗口”描述要求现金缓冲、恢复能力和可迁移技能，实际只检查年龄、现金大于 0 和精神大于 0。

“不卷也能活”没有检查加班历史，只检查三年后健康和精神状态。

“普通结局：工具人的一生”仍然把普通职业人生描述为缺乏意义，与本轮价值观整改不一致。

成就条件必须能够从名字和描述直接推导，不能只更换展示文案。

### FOLLOWUP-014：分支历史治理仍未解决

**级别：P1**

`fix/v6-audit-remediation-p0` 基于 `feature/realworld-data-import`，而 V6 分支仍与默认分支 `master` 没有共同祖先。

建议尽快决定：

- V6 是否成为新的默认产品线；
- 旧版是否归档为 tag 或 legacy 分支；
- 是否从 `master` 创建具有共同历史的 V6 重构分支；
- 部署和回滚使用哪条分支。

在此问题明确前，不建议把修复分支直接宣称为可发布主线。

## 6. 测试与发布门槛复审

### 6.1 已有进步

- 自动策略从一种扩展到八种；
- 开始统计破产、燃尽、健康危机、失业时长、首份 Offer、净资产和结局分布；
- 增加确定性重放；
- 增加百分制范围、家庭状态和结局条件检查；
- 依赖版本已经锁定。

### 6.2 当前测试不能证明的内容

1. 批量策略仍然每月调用 `applyAction()`，没有测试多行动计划；
2. “结局可达性”直接修改状态使条件为真，不验证从初始状态是否存在合法行动路径；
3. 没有测试危机恢复成功和复发；
4. 没有测试事件链计数与冷却互不覆盖；
5. 没有测试公司薪资和晋升参数真实参与公式；
6. 没有测试玩家从 UI 使用月度计划；
7. 没有测试第二个项目能否创建；
8. 没有测试证据来源分类是否与真实来源类型一致。

### 6.3 发布阈值过于宽松

当前阈值允许：

- 破产率 85%；
- 燃尽率 70%；
- 健康危机率 70%；
- 单一行动占比 95%。

这些阈值只能发现系统完全失控，不能证明游戏平衡。`minDeterministicReplayRate` 还没有真正参与发布判断。

建议分三类门槛：

| 类型 | 示例 |
|---|---|
| 硬不变量 | 资产守恒、字段范围、无非法状态、存档可迁移 |
| 分布门槛 | 各路线结局率、失业周期、危机率、收入分布 |
| 玩法门槛 | 无全局统治行动、每种价值路线可行、重复策略明显受惩罚 |

结局可达性应通过受约束搜索或大量合法策略轨迹验证，不能直接构造终态。

## 10. 最终判断

本轮整改最值得肯定的地方，是把首轮审计中的抽象建议转化成了清晰的模块和类型。金额、投资、求职计数、家庭成员状态和依赖锁定等底层问题也已经得到实质修复。

但工程中出现了新的风险：**把“定义了系统”误认为“交付了系统”。**

- 月度计划存在于引擎，不存在于玩家操作；
- 职业路线存在于配置，不存在于人生路径；
- 危机存在恢复字段，不存在恢复过程；
- 价值目标存在权重，不存在玩家选择；
- 证据存在枚举，不存在可靠追溯；
- 可达性存在测试，不存在合法轨迹证明。

下一轮应停止继续增加新名词和新配置，优先把已经创建的系统接通、闭环并让玩家看得见。完成 FOLLOWUP-001 至 FOLLOWUP-005 后，V6 才真正具备从高级原型进入内部测试版本的条件。
